# System Patterns: TMB Import Dockerization

## System Architecture:

The `tmb-import` application transitions from a Windows-native scheduled task to a Docker-containerized service orchestrated by Docker Compose on an Ubuntu server. This introduces a client-server architecture where the Docker host runs the containerized application.

```mermaid
graph TD
    A[Ubuntu Server] --> B(Docker Engine)
    B --> C{tmb-import Docker Container}
    C --> D[Node.js Application (src/app.js)]
    C --> E[Git Client]
    D -- Reads config --> F[config.yaml]
    D -- Writes data --> G[temp/tmb-data.json & temp/tmb-items.csv]
    E -- Clones/Pulls/Pushes --> H[loot.git Repository (GitHub Pages)]
    C -- Scheduled by --> I[Cron Job]
```

## Key Technical Decisions:

- **Containerization:** Using Docker to encapsulate the application and its dependencies (Node.js, `puppeteer`, `git`). This ensures a consistent runtime environment across different hosts.
- **Orchestration with Docker Compose:** Simplifies the definition and management of the `tmb-import` service, including volume mounts for data persistence and configuration.
- **Configuration Management with YAML:** Centralizes application settings in `config.yaml`, providing a human-readable and version-controllable (via `config.example.yaml`) format while keeping sensitive data out of version control.
- **Entrypoint Script for Workflow:** A dedicated `entrypoint.sh` script manages the sequential execution of Git operations and the Node.js application, mimicking the original `task.bat` workflow within the container context.
- **Volume Mounts for Data:** Persistent volumes (`./repo` and `./temp`) are mounted to allow the container to interact with the host filesystem for Git repository storage and temporary data.
- **Cron for Scheduling:** Leveraging the native Linux cron utility for scheduling regular execution of the Docker Compose service, aligning with standard Linux practices.

## Design Patterns in Use:

- **Containerization Pattern:** Encapsulating the application and its environment into an isolated Docker container.
- **Configuration Externalization Pattern:** Separating configuration (e.g., `config.yaml`) from the application code for flexibility and security.
- **Entrypoint Script Pattern:** Using a shell script as the container's entrypoint to manage the application's startup lifecycle, including setup and execution of multiple steps.
- **Volume Mount Pattern:** Persisting data generated or modified by the container on the host filesystem.

## Component Relationships:

- **`Dockerfile`:** Builds the `tmb-import` Docker image.
- **`docker-compose.yml`:** Defines how to run the `tmb-import` container, referencing the Dockerfile and specifying volume mounts.
- **`config.yaml`:** Provides runtime configuration to `src/app.js`.
- **`src/app.js`:** The core Node.js application logic, now reading configuration from `config.yaml`.
- **`entrypoint.sh`:** The orchestrator script inside the container, executing Git commands and then `src/app.js`.
- **`repo` directory (host/mounted volume):** Stores the `loot.git` repository, allowing persistent Git operations.
- **`temp` directory (host/mounted volume):** Stores temporary data (`tmb-data.json`, `tmb-items.csv`) generated by `src/app.js` before being copied to `repo/assets`.

## Critical Implementation Paths:

1.  **Container Build:** `Dockerfile` -> `npm install` -> Copy application files.
2.  **Application Execution Flow:** `cron` -> `docker compose run` -> `entrypoint.sh` -> `git clone/pull` -> `cd repo` -> `git checkout gh-pages` -> `cp temp/*.json repo/assets/` -> `git add/commit/push` -> `node src/app.js`.
3.  **Configuration Loading:** `src/app.js` -> `js-yaml` -> `config.yaml`.
4.  **Data Flow:** `src/app.js` (downloads) -> `temp/tmb-data.json` -> `entrypoint.sh` (copies) -> `repo/assets/tmb-data.json` -> `git add/commit/push`.
5.  **Git Authentication:** Requires proper setup of SSH keys (recommended) or HTTPS with PAT for `git push` operations within the container to the `loot.git` repository.
